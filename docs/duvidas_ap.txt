1) PERGUNTAS SOBRE A IDEIA DO PROJETO
============================================================

1. O que é o OrbitGuard Climate?

Resposta:
O OrbitGuard Climate é uma API para simular o monitoramento de áreas com risco climático. A ideia é trabalhar com dados simulados de satélite, como temperatura, umidade, chuva e índice de vegetação, para gerar alertas climáticos.

Resposta mais completa:
O projeto não consome um satélite real. Ele simula esse cenário para representar uma solução ligada à economia espacial. A API organiza áreas monitoradas, leituras simuladas e alertas, tudo persistido em um banco MySQL dentro de container.


2. Qual problema sua solução tenta resolver?

Resposta:
Ela tenta ajudar no monitoramento de regiões vulneráveis a problemas climáticos, como seca, calor extremo ou risco ambiental.

Resposta mais completa:
O objetivo é mostrar como dados observacionais, que poderiam vir de satélites, podem ser usados para apoiar decisões. Por exemplo, uma área com temperatura alta, baixa umidade e baixo índice de vegetação pode gerar um alerta de risco de seca.


3. Por que esse projeto tem relação com economia espacial?

Resposta:
Porque ele usa a ideia de dados de observação da Terra, que são um dos usos mais importantes da tecnologia espacial.

Resposta mais completa:
Mesmo usando dados simulados, o conceito do projeto se baseia em informações que poderiam vir de satélites. Esses dados podem ser aplicados em agricultura, defesa civil, meio ambiente e monitoramento climático.


4. Os dados são reais ou simulados?

Resposta:
São simulados.

Resposta boa:
Eu deixei os dados simulados porque o foco da disciplina era DevOps, Docker, nuvem e arquitetura. O projeto representa como uma solução real poderia usar dados de satélite, mas aqui o objetivo foi criar uma API funcional, conteinerizada e em nuvem.


5. O que significa “área monitorada” no sistema?

Resposta:
É uma região cadastrada no sistema que pode receber leituras climáticas e alertas.

Exemplo:
Uma área pode ser “Ribeirão Preto - SP”, com latitude, longitude e nível de risco. Depois ela pode receber leituras simuladas e alertas relacionados.

2) PERGUNTAS SOBRE ARQUITETURA
============================================================

6. Explique a arquitetura da sua solução.

Resposta:
A arquitetura tem dois containers principais: um container da API .NET e outro container do MySQL. Eles se comunicam por uma rede Docker e o banco usa volume para persistência.

Resposta mais completa:
A API recebe as requisições pelo Swagger e pelos endpoints REST. Ela usa Entity Framework para acessar o MySQL. O MySQL fica em um container separado. Os dois containers estão na rede orbitguard-network, e o banco usa o volume orbitguard-mysql-data para manter os dados.


7. Por que você separou API e banco em containers diferentes?

Resposta:
Para separar responsabilidades.

Resposta boa:
A API é responsável pelas regras, endpoints e comunicação com o usuário. O MySQL é responsável por armazenar os dados. Separar em containers deixa a solução mais organizada, mais fácil de manter e mais parecida com um ambiente real.


8. O que aconteceria se a API e o banco estivessem no mesmo container?

Resposta:
Funcionaria de forma menos organizada e seria uma má prática.

Resposta boa:
Um container deve ter uma responsabilidade principal. Se eu colocasse API e banco no mesmo container, ficaria mais difícil atualizar, escalar, diagnosticar erro e manter persistência. Por isso separei em dois serviços.


9. Como a API encontra o banco?

Resposta:
Ela encontra o banco pelo nome do container/serviço dentro da rede Docker.

Resposta boa:
No Docker Compose, a API usa a connection string apontando para orbitguard-db-rm93032. Dentro da rede Docker, esse nome funciona como endereço do banco. Por isso a API não usa localhost para acessar o MySQL.


10. Por que não usar localhost na connection string da API?

Resposta:
Porque dentro do container da API, localhost seria o próprio container da API, não o banco.

Resposta boa:
Essa é uma dúvida importante. Cada container tem seu próprio ambiente. Se a API tentar acessar localhost, ela procura o MySQL dentro dela mesma. Como o banco está em outro container, eu uso o nome do serviço/container na rede Docker.


11. O que é a rede orbitguard-network?

Resposta:
É a rede Docker criada para permitir a comunicação entre a API e o banco.

Resposta mais completa:
Com essa rede, os containers conseguem se enxergar pelo nome. Isso permite que a API se conecte ao banco de forma interna, sem depender do IP do container, que pode mudar.


12. O que é o volume orbitguard-mysql-data?

Resposta:
É o volume que guarda os dados do MySQL fora do ciclo de vida do container.

Resposta boa:
Sem volume, se eu removesse o container do banco, poderia perder os dados. Com volume, os dados ficam persistidos no Docker e podem continuar existindo mesmo se o container for recriado.


13. O que significa persistência de dados no seu projeto?

Resposta:
Significa que os dados do banco continuam salvos mesmo se o container for parado ou recriado.

Resposta boa:
O MySQL grava os dados no volume Docker. Então a aplicação pode ser reiniciada sem perder automaticamente as áreas, leituras e alertas já cadastrados.


14. A aplicação está rodando em localhost ou em nuvem?

Resposta:
Está rodando em nuvem.

Resposta boa:
A API está em uma VM Linux na DigitalOcean. Quando eu acesso o Swagger pelo IP público 146.190.218.162 na porta 8080, estou acessando a aplicação pela internet, não pelo localhost do meu computador.


15. Como você prova que está em nuvem?

Resposta:
Eu provo de três formas: acesso pelo IP público, conexão SSH/Web Console na VM e docker ps mostrando os containers rodando dentro da máquina em nuvem.

Resposta boa:
Além disso, o Swagger abre pelo endereço público da VM. Isso mostra que a API está publicada fora da minha máquina local.

3) PERGUNTAS SOBRE DOCKER
============================================================

16. O que é Docker?

Resposta:
Docker é uma tecnologia para empacotar aplicações em containers.

Resposta boa:
Com Docker, eu consigo colocar a aplicação e suas dependências em um ambiente padronizado. Assim, o projeto roda de forma parecida no meu computador e na VM em nuvem.


17. O que é um container?

Resposta:
É um ambiente isolado onde um serviço roda com suas dependências.

Resposta boa:
No meu projeto, existe um container para a API e outro para o MySQL. Cada um roda isolado, mas eles se comunicam pela rede Docker.


18. O que é uma imagem Docker?

Resposta:
Imagem é o modelo usado para criar containers.

Resposta boa:
No meu projeto, a imagem da API é construída a partir do Dockerfile. Já o MySQL usa a imagem oficial mysql:8.0.


19. Qual a diferença entre imagem e container?

Resposta:
Imagem é o molde. Container é a execução desse molde.

Exemplo:
A imagem orbitguard-api-rm93032:1.0 é usada para criar o container orbitguard-api-rm93032.


20. Para que serve o Dockerfile?

Resposta:
Serve para ensinar o Docker a construir a imagem da minha API.

Resposta boa:
No Dockerfile eu defino a imagem base do .NET, copio o projeto, restauro dependências, publico a aplicação, crio um usuário não-root, exponho a porta 8080 e defino o comando de inicialização da API.


21. Por que seu Dockerfile tem duas fases?

Resposta:
Para separar build e execução.

Resposta boa:
A primeira fase usa o SDK do .NET para restaurar, compilar e publicar o projeto. A segunda fase usa a imagem runtime do ASP.NET, que é mais adequada para executar a aplicação. Isso deixa a imagem final mais limpa.


22. O que é multi-stage build?

Resposta:
É quando o Dockerfile tem mais de uma etapa de construção.

Resposta boa:
No meu caso, uma etapa compila a aplicação e a outra executa só o resultado final publicado. Isso evita levar ferramentas de build para a imagem final.


23. Por que você criou o usuário appuser?

Resposta:
Para a API não rodar como root dentro do container.

Resposta boa:
Rodar como root aumenta o risco de segurança. Mesmo sendo um projeto acadêmico, configurei o appuser para mostrar uma prática melhor de segurança no Dockerfile.


24. Como você prova que a API não está rodando como root?

Resposta boa:

27. O que é Docker Compose?

Resposta:

Resposta boa:
No meu projeto, o docker-compose.yml configura a API, o banco MySQL, rede, volume, portas, variáveis de ambiente e dependência entre os containers.

Com docker run eu teria que digitar comandos longos para API, banco, rede, volume e variáveis. Com Compose, tudo fica documentado e repetível no arquivo docker-compose.yml.

31. O que é healthcheck no banco?

É uma verificação para saber se o MySQL está pronto para receber conexões.

Resposta boa:
O Compose usa o healthcheck para esperar o banco ficar saudável antes de subir a API. Isso evita a API tentar conectar antes do MySQL estar pronto.

Resposta boa:

Na apresentação, ele prova que tanto a API quanto o MySQL estão rodando na VM em nuvem, com as portas publicadas.

39. O que a tabela satellite_readings guarda?

Resposta:
Uma área monitorada pode ter várias leituras de satélite e vários alertas climáticos. Um alerta pode estar ligado a uma leitura específica.

Resposta mais completa:
monitored_areas tem relação um para muitos com satellite_readings.
monitored_areas tem relação um para muitos com climate_alerts.
satellite_readings pode se relacionar com climate_alerts.

Resposta:
Porque ele prova que os dados estão relacionados no banco.

Resposta boa:
O JOIN junta a área, a leitura e o alerta em uma única consulta. Isso mostra que a estrutura do banco não é só um cadastro isolado, mas tem relacionamento entre entidades.

Resposta boa:

Resposta:
O volume mantém os dados enquanto ele existir. Mas se eu deletar registros com comando SQL ou remover o volume, os dados somem.

Resposta boa:
Persistência não impede exclusão lógica dos dados. Ela só garante que o armazenamento sobreviva ao ciclo de vida do container.


45. O que é uma chave estrangeira?

Resposta:
É um campo que liga uma tabela à outra.

Exemplo:
satellite_readings tem monitored_area_id para indicar a qual área aquela leitura pertence.


46. O que é CRUD?

Resposta:
CRUD significa Create, Read, Update e Delete.

Resposta boa:
No projeto, os controllers permitem criar, listar, atualizar e excluir registros das entidades principais da API.


47. O banco foi criado manualmente ou pela aplicação?

Resposta:
A estrutura foi gerada pela aplicação usando Entity Framework/migrations.

Resposta segura:
Eu modelei as entidades na API, configurei o contexto do Entity Framework e usei migration para criar as tabelas no MySQL.


48. O que é Entity Framework?

Resposta:
É uma ferramenta ORM do .NET que permite trabalhar com banco usando classes e objetos.

Resposta boa:
Em vez de escrever SQL para tudo, eu modelo as entidades em C# e o Entity Framework faz a comunicação com o banco MySQL.


49. O que é uma migration?

Resposta:
É uma versão da estrutura do banco gerada a partir das entidades do código.

Resposta boa:
A migration registra mudanças como criação de tabelas, campos e relacionamentos. Ela ajuda a manter o banco alinhado com o código.

70. Me explica o fluxo completo de uma requisição no seu sistema.

Resposta boa:

O Docker Compose, porque ele organiza os dois containers, a rede, o volume, as portas e as variáveis de ambiente. Sem ele, eu teria que configurar tudo manualmente.

Porque dentro do container da API, localhost é a própria API. O banco está em outro container, então preciso usar o nome dele na rede Docker.
