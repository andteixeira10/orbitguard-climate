1. ABERTURA E IDEIA DO PROJETO
============================================================

FALA:

Boa tarde, eu sou o André Teixeira Alves Filho, RM 93032.

O meu projeto se chama OrbitGuard Climate.

A ideia dele é simular uma solução de monitoramento de áreas de risco climático usando dados simulados de satélite.

Na prática, a API trabalha com três informações principais:
áreas monitoradas, leituras climáticas simuladas e alertas climáticos.

Por exemplo: uma área pode receber uma leitura com temperatura alta, baixa umidade, pouca chuva e índice de vegetação baixo.
Com esses dados, o sistema pode registrar um alerta de risco climático.

A solução foi feita com uma API em .NET, banco MySQL e Docker.
E o principal ponto da apresentação é mostrar que tudo está rodando em uma VM Linux em nuvem.

4. ARQUITETURA DA SOLUÇÃO
============================================================

FALA:

A arquitetura tem dois containers principais.

O primeiro container é a API .NET, chamada orbitguard-api-rm93032.

O segundo container é o banco MySQL, chamado orbitguard-db-rm93032.

Esses dois containers ficam na mesma rede Docker, chamada orbitguard-network.

Um ponto importante é que a API não chama o banco usando localhost.
Ela chama o banco pelo nome do container: orbitguard-db-rm93032.

Isso funciona porque os containers estão na mesma rede Docker.

O banco usa um volume Docker para persistir os dados.
Então, mesmo que o container seja recriado, os dados do MySQL ficam salvos no volume.

A API expõe a porta 8080 e o MySQL usa a porta 3306.

7. BANCO MYSQL E RELACIONAMENTO DOS DADOS
============================================================

TELA:
Voltar para o terminal.

COMANDO 1 - MOSTRAR TABELAS:

docker exec -e MYSQL_PWD=orbitguard_pass -it orbitguard-db-rm93032 mysql -uorbitguard_user -t -e "USE orbitguard_db; SHOW TABLES;"

FALA:

Agora eu estou entrando no container do MySQL e executando um comando SQL direto no banco.

O SHOW TABLES mostra as tabelas da aplicação.

As principais são:
monitored_areas, que guarda as áreas monitoradas;
satellite_readings, que guarda as leituras simuladas de satélite;
climate_alerts, que guarda os alertas climáticos.

COMANDO 2 - SELECT COM JOIN:

docker exec -e MYSQL_PWD=orbitguard_pass -it orbitguard-db-rm93032 mysql -uorbitguard_user -t -e "USE orbitguard_db; SELECT a.name AS area_name, a.location, r.temperature, r.humidity, r.rainfall, r.vegetation_index, c.alert_type, c.severity, c.status FROM climate_alerts c JOIN monitored_areas a ON a.id = c.monitored_area_id LEFT JOIN satellite_readings r ON r.id = c.satellite_reading_id;"

FALA:

Agora eu faço um SELECT com JOIN.

Esse comando junta três informações:
a área monitorada,
a leitura climática simulada,
e o alerta gerado para aquela área.

Isso mostra que o banco não tem só tabelas soltas.
Os dados têm relacionamento entre si.

A área se relaciona com as leituras e também com os alertas.
E o alerta pode estar relacionado com uma leitura específica.

8. PONTOS IMPORTANTES DO DOCKER
============================================================

COMANDO 1 - USUÁRIO NÃO-ROOT:

docker container exec orbitguard-api-rm93032 whoami

FALA:

Esse comando mostra com qual usuário a API está rodando dentro do container.

O retorno esperado é appuser.

Isso é importante porque no Dockerfile eu configurei a aplicação para não rodar como root.
É uma prática melhor de segurança.

COMANDO 2 - VOLUME:

docker volume ls

FALA:

Aqui eu mostro os volumes Docker.

O volume do projeto é usado pelo MySQL para guardar os dados.

A ideia do volume é separar os dados do banco do ciclo de vida do container.
Então, se o container for recriado, os dados continuam no volume.

10. PLANO DE EMERGÊNCIA
============================================================

SE O POWERSHELL NÃO CONECTAR:

Usar DigitalOcean > Droplet > Web Console

SE OS CONTAINERS NÃO APARECEREM:

cd /root/orbitguard-climate
docker-compose up -d
docker ps

SE O SWAGGER NÃO ABRIR:

docker ps
curl http://localhost:8080/
docker logs --tail 50 orbitguard-api-rm93032

NÃO RODAR NA APRESENTAÇÃO:

docker-compose down -v

Esse comando apaga o volume se usado com -v.
Não usar.

Também evitar rebuild na apresentação, a não ser que seja emergência:

docker-compose up -d --build
